# 社区互助平台 PRD（逆向版）

## 1. 文档说明

- **文档目的**：基于现有前后端工程，逆向提炼产品需求与业务逻辑，形成可用于论文、评审和后续迭代的产品需求文档。
- **范围**：移动端（居民/志愿者）+ 管理端（社区管理员/超级管理员）。
- **约束**：仅描述已实现或工程中明确可见的能力，不虚构功能，不包含代码实现细节。

---

## 2. 产品定位与目标

### 2.1 产品定位

一个面向社区场景的互助服务平台，连接“有需求的居民”和“可提供帮助的志愿者”，并提供社区管理与平台治理能力。

### 2.2 产品目标

1. 打通社区互助全流程：发布需求、审核、认领、履约、确认、评价。
2. 提升供需匹配效率与可追踪性。
3. 通过管理端看板、审计、导出备份、系统配置保障平台可运营与可治理。
4. 引入 AI 辅助咨询能力，降低用户决策与操作门槛。

### 2.3 核心价值

- **对居民**：更快找到帮助，流程透明可追踪。  
- **对志愿者**：任务获取清晰，贡献可沉淀（含时间币结算线索）。  
- **对管理者**：有数据、有规则、有审计，便于日常运营与风控。

## 3. 用户角色与 RABC

> 注：本项目中的“RABC”按产品管理视角定义为：  
> **R（Role）角色**、**A（Action）动作权限**、**B（Boundary）数据边界**、**C（Control）控制策略**。

## 3.1 角色定义

- **普通用户（USER）**：移动端主角色，细分为居民与志愿者（身份由 `identityType` 区分）。  
- **社区管理员（COMMUNITY_ADMIN）**：管理辖区业务。  
- **超级管理员（SUPER_ADMIN）**：全局管理与系统治理。

## 3.2 RABC 权限矩阵


| 角色（R）    | 动作权限（A）                              | 数据边界（B）                  | 控制策略（C）                 |
| -------- | ------------------------------------ | ------------------------ | ----------------------- |
| 普通用户-居民  | 发布需求、查看需求进度、确认完成、评价、查看通知、加入社区、编辑个人资料 | 仅本人相关数据 + 所属社区公开需求       | 登录态校验；仅可操作本人发起/参与的单据    |
| 普通用户-志愿者 | 浏览并认领需求、提交完成、查看收到评价、查看通知             | 仅本人相关数据 + 所属社区公开需求       | 登录态校验；流程状态前置校验（未认领不可完成） |
| 社区管理员    | 需求审核、辖区监控、辖区看板、辖区导出、提醒/干预            | 仅绑定 `community_id` 的辖区数据 | 角色鉴权 + 社区范围过滤           |
| 超级管理员    | 全局用户管理、全局看板、审计日志、备份恢复、系统配置、全局导出      | 全平台数据                    | 角色鉴权 + 高风险操作留痕（审计/备份）   |


## 3.3 控制规则要点

1. **角色鉴权**：接口层按角色声明访问权限（用户、社区管理员、超级管理员）。
2. **数据边界**：管理员能力按是否超级管理员决定是否套用 `community_id` 过滤。
3. **流程控制**：需求、认领、完成、确认、评价均依赖前置状态，防止越权或越序操作。
4. **登录与引导控制**：已登录但未绑定社区时，强制引导至加入社区流程。

## 4. 需求分析

## 4.1 业务痛点

1. 社区互助需求分散，难以及时匹配到可服务志愿者。
2. 需求从发起到完成缺少统一流程，状态不可视。
3. 管理端对业务质量、风险与效率缺乏统一看板。
4. 缺少日志审计与备份机制，治理和追溯能力弱。

## 4.2 需求拆解

- **用户侧需求**：简单发布、快速认领、状态透明、结果可评价。  
- **管理员需求**：可审核、可监控、可分析、可导出、可配置。  
- **系统侧需求**：权限可控、数据可追溯、流程可闭环、配置可迭代。

## 4.3 业务闭环目标

`需求发布 -> 管理审核 -> 志愿者认领 -> 服务完成提交 -> 需求方确认 -> 评价沉淀 -> 数据看板反馈`

## 5. 功能需求与实现逻辑（不含代码）

## 5.1 移动端功能

### 5.1.1 账号与身份

- 登录、注册、验证码、找回/修改密码、用户引导信息维护。  
- 注册后支持居民/志愿者身份分化，影响后续任务行为与展示内容。

### 5.1.2 首页与个人中心

- 展示社区、积分、互助统计、勋章、快捷入口。  
- 用户资料可编辑（头像、地址、技能等），支撑后续匹配与服务可信度。

### 5.1.3 服务大厅与任务页

- 展示服务汇总、进行中任务、发布入口、认领入口。  
- 支持“我参与的帮助”和“我发起的求助”双视角切换。

### 5.1.4 需求发布

- 用户填写服务类型、紧急程度、地址、联系人、描述。  
- 提交后进入待审核状态，审核通过后进入可被认领池。

### 5.1.5 需求认领与履约

- 志愿者浏览已发布需求并认领。  
- 志愿者提交完成信息（服务时长/说明），需求方确认后结算完成。  
- 若有争议可发起申诉，管理员可人工介入。

### 5.1.6 评价反馈

- 需求方在完成后对志愿者评分与评价。  
- 志愿者可查看“我收到的评价”，需求方可查看“我发出的评价历史”。

### 5.1.7 通知与公告

- 提供通知列表、未读计数、标记已读、全部已读。  
- 公告、消息提醒与业务状态变化联动。

### 5.1.8 社区加入/改绑

- 用户可通过邀请码校验并加入社区。  
- 加入成功后覆盖绑定社区，用于后续数据边界与任务归属。

### 5.1.9 AI 对话

- 提供 AI 问答入口，支持跨页面携带问题进入。  
- 用于咨询建议、任务理解与操作辅助。

## 5.2 管理端功能

### 5.2.1 仪表盘

- 统计总览、覆盖率、趋势、榜单、月度匹配率等看板指标。  
- 社区管理员看辖区，超级管理员看全局。

### 5.2.2 需求审核与监控

- 对需求执行通过/驳回。  
- 查看服务监控列表（如超时、进行中关键状态）。

### 5.2.3 用户管理

- 列表查询、创建、编辑、禁用/启用、删除。  
- 支持按角色、状态、社区筛选。

### 5.2.4 审计日志

- 按时间、模块、风险等级、操作结果检索。  
- 查看日志详情，支撑运营追责与安全审计。

### 5.2.5 数据导出与备份

- 模块化导出（支持格式与时间范围）。  
- 备份运行、计划策略、恢复、历史下载与删除。

### 5.2.6 系统配置

- 基础参数、通知模板、预警规则、AI 参数分区管理。  
- 支持 AI 配置连通性测试。

### 5.2.7 内容与社区运营

- 轮播图管理、公告管理、邀请码管理、社区选项管理。  
- 支撑社区日常运营传播与用户触达。

## 6. 核心业务逻辑

## 6.1 需求生命周期（主流程）

1. 居民发布需求（创建）。
2. 管理员审核（通过/驳回）。
3. 通过后进入可认领池。
4. 志愿者认领后进入进行中。
5. 志愿者提交完成，进入待确认。
6. 需求方确认后完成结算。
7. 进入评价阶段，沉淀服务口碑数据。

## 6.2 异常与分支逻辑

- 审核驳回：需求终止或修改后重发。  
- 履约争议：需求方申诉，管理员介入处理。  
- 未登录/越权：统一鉴权拦截并返回错误或跳转。  
- 待确认超时：志愿者提交完成后，若需求方在约定窗口内无异议，进入自动完成策略。  
- 社区未绑定：用户登录后若无社区信息，先引导加入社区再进入业务流程。

## 6.3 时间币结算逻辑（已体现业务意图）

- 志愿者提交完成不直接结算。  
- 需求方确认完成后触发“时间币已结算”反馈。  
- 该机制可作为后续“时间银行”激励体系扩展基础。

## 6.4 数据回流逻辑

- 评价、完成、审核、申诉等行为会回流到仪表盘统计与审计日志。  
- 管理端通过导出、看板、日志形成“业务执行—治理反馈”闭环。

## 6.5 志愿者需求大厅展示规则【新增】

- **规则定位**：用于志愿者在需求大厅查看“可认领帖子”时的筛选与排序。  
- **第一层（硬过滤）**：仅展示“已发布”状态需求；排除本人发布需求；限定同社区需求；过滤已删除数据。  
- **第二层（技能匹配）**：基于志愿者技能标签与需求标签/服务类型做匹配，命中任一标签视为技能匹配。  
- **第三层（排序）**：按紧急程度降序、发布时间降序、技能匹配优先、创建时间降序进行综合排序。  
- **业务目标**：保证“能接、该接、优先接”，在合规边界内提升认领效率。

## 7. 非功能需求

1. **安全性**：基于角色的接口访问控制，关键操作鉴权。
2. **可追溯性**：关键管理行为记录审计日志。
3. **可运维性**：支持数据导出、备份恢复、计划策略。
4. **可配置性**：规则和模板可后台动态配置。
5. **可扩展性**：AI 配置、时间币结算、社区运营模块具备扩展空间。

## 8. 作图约束与可视化输入

> 本节用于向画图 AI 传递统一口径，减少图形偏差。

## 8.1 系统边界

- **系统内**：移动端、管理端、后端 API、数据库、审计模块、备份模块、配置中心。  
- **系统外**：AI 服务（外部模型）、短信/通知网关（外部通道）。  
- **边界规则**：用户请求必须经后端 API 执行业务校验后落库，外部服务不直接写库。

## 8.2 主流程状态枚举（需求单）

- `CREATED`：待审核  
- `APPROVED`：已发布  
- `IN_PROGRESS`：进行中  
- `PENDING_CONFIRM`：待确认  
- `COMPLETED`：已完成  
- `REJECTED / CANCELLED`：驳回或取消（异常终态）

## 8.3 异常分支（出图必须体现）

1. 审核驳回分支（含驳回原因回传）。
2. 申诉分支（需求方发起，管理员介入）。
3. 待确认超时分支（自动完成）。
4. 社区未绑定分支（强制引导加入社区）。
5. 未授权分支（鉴权失败直接拦截）。

## 8.4 ER 图最小实体集

- 用户（User）  
- 社区（Community/Region）  
- 服务需求（ServiceRequest）  
- 服务认领（ServiceClaim）  
- 服务评价（ServiceEvaluation）  
- 通知（Notification）  
- 邀请码（InviteCode）  
- 审计日志（AuditLog）  
- 备份记录（BackupRecord）

## 8.5 系统架构图分层约束

1. 接入层：移动端、管理端。
2. 接口层：认证、需求、认领、评价、通知、看板、管理接口。
3. 业务层：流程编排、权限校验、规则引擎、时间币结算。
4. 数据层：业务库、日志库、配置存储、备份存储。
5. 治理层：审计、导出、备份恢复、系统配置。
6. 外部服务层：AI 服务、短信/通知通道。

## 9. 当前边界与后续优化方向

## 9.1 当前边界（从工程逆向观察）

- 通知/会话部分在移动端存在演示化内容，实时通信能力可继续增强。可以仅完成社区信息和AI助手信息，居民-居民通信要求可以不做。  
- 备份恢复中存在“恢复任务已提交（模拟）”语义，后续可补充完整恢复编排。  
- 部分业务状态文案已具备，但统一状态字典与可视化流程可继续强化。  
- 需求订单展示的卡片要 展示必要的信息，求助人、志愿者、手机号、地址等。给顶单卡片step进度条做成组件单独的卡片，优化UI
- 项目中的志愿积分和时间币的地方不统一，全换成时间币的概念。志愿者完成一单+5个时间币。

## 9.2 迭代建议

1. 强化智能推荐：基于用户画像与历史履约实现需求推荐。
2. 强化风控规则：结合审计日志与异常行为建立预警策略。
3. 时间银行升级：将时间币作为长期激励资产，完善跨代传递与权益兑换机制。
4. 用户增长运营：完善消息触达、活动模块与社区运营工具链。

---

## 10. 验收口径（PRD 视角）

- 用户端核心闭环可全流程跑通：发布、审核、认领、完成、确认、评价。  
- 管理端核心治理能力可用：看板、审核、用户管理、日志审计、导出备份、系统配置。  
- 权限与数据边界符合 RABC 设计：角色可控、范围可控、操作可追溯。
